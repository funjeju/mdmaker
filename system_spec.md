# 대화형 MD 파일 생성기 (Interactive MD File Generator) 기획서

## 전체 정의: “대화형 MD 파일 생성기”
비개발자가 AI와 대화만으로 웹앱 개발 명세를 만들고, 최종적으로 실제 개발 도구에 즉시 투입 가능한 프로젝트 구조를 생성하는 시스템이다. 목표는 아이디어를 개발 가능한 상태까지 추상화 없이 연결하는 것이다. 사용자는 기술 용어를 몰라도 되며, AI가 쉬운 질문을 내부적으로 개발 언어와 구조로 변환한다.

## 핵심 철학: 비개발자 기준 설계
모든 질문은 개발자 언어가 아니라 생활 언어로 바뀌어야 한다. 예를 들어 인증 구조를 묻지 않고 “혼자 쓰나요, 여러 사람이 쓰나요?”처럼 질문한다. 사용자는 목적과 아이디어만 설명하면 되고, 기술 선택은 AI가 자동 변환한다. 질문마다 “왜 이 질문을 하는지”를 쉬운 말로 설명해 불안감을 줄인다.

---

## 1단계: 공통 사전 질문 시스템
모든 웹앱에서 공통으로 필요한 최소 정보를 가장 먼저 확보한다. 질문은 매우 추상적이고 쉬운 언어로 시작한다. 무엇을 만들고 싶은지, 누구를 위한 서비스인지, 혼자 쓰는지 여러 사람이 쓰는지, 로그인 여부, 모바일/PC 중심 여부, 데이터 저장 필요성, 외부 API 사용 여부, AI 기능 필요 여부, 결제 여부, 사용할 개발 툴, 배포 여부, 민감 데이터 존재 여부, 속도 우선인지 완성도 우선인지 등을 질문한다. 이 단계에서 CORE.md 초안을 생성할 최소 정보가 확보된다.

## 2단계: 개발 환경 및 방식 결정
사용자 답변을 기반으로 프로젝트 성격을 자동 확정한다. 예를 들어 혼자 쓰는 도구이며 로그인과 서버가 필요 없으면 로컬 중심 구조를 추천하고, 여러 사용자가 접근하는 서비스면 DB와 인증 구조를 자동 제안한다. 사용자는 기술을 몰라도 되며 AI가 내부적으로 최적 구조를 선택한다.

## 3단계: API Vault(공통 API 키 창고)
프로젝트마다 API를 새로 등록하는 것이 아니라 중앙 API 창고를 만든다. 모든 프로젝트는 공통 저장소에서 키를 참조한다. 프로젝트 생성 시 필요한 API를 자동 탐색하여 env.local을 생성한다. 키가 없으면 “나중에 넣을까요?”처럼 안내한다. 또한 글로벌 키와 프로젝트별 override 구조를 둬 특정 프로젝트에서만 다른 키를 사용할 수 있게 한다. 키 존재 여부 검사와 자동 연결 기능은 필수다.

## 4단계: 아이디어 구체화 인터뷰
가장 핵심 단계다. 사용자의 모호한 아이디어를 MD 명세 수준까지 구체화한다. AI는 아이디어를 요약하고 쉬운 질문을 단계적으로 던진다. 첫 화면에서 무엇을 하는지, 핵심 기능은 무엇인지, 사람들이 왜 계속 쓰는지, 실패 상황은 어떻게 처리해야 하는지 등을 묻는다. 이를 통해 PRODUCT, USER_FLOW, FEATURE_SPEC 품질을 높인다.

## 5단계: 질문 공정률 시스템
단순 진행률이 아니라 문서 생성 준비도를 시각화한다. 예를 들어 CORE 준비도, FEATURE 준비도, USER_FLOW 준비도를 따로 보여준다. 사용자는 전체 명세 완성도를 확인하며 중간 이탈을 줄일 수 있다.

## 6단계: CORE.md 우선 생성
질문 종료 직후 CORE.md를 가장 먼저 생성한다. CORE는 모든 문서의 기준점이며 이후 생성될 문서들의 참조 루트가 된다. CORE 생성 시점부터 전체 문서 트리가 열린다.

## 7단계: MD 트리 기반 생성 UI
좌측에는 문서 트리, 우측에는 문서 편집 영역을 둔다. 각 MD는 생성하기, 수정하기, AI에게 다시 묻기, 부족한 정보 질문받기, 저장 기능을 가진다. 문서 생성 시 선행 문서를 자동 참조한다. 예를 들어 FEATURE_SPEC 생성 시 CORE와 PRODUCT를 자동 참조한다.

## 8단계: 문서 생성 순서 강제
문서 품질을 위해 생성 순서를 강제한다. 순서는 CORE → PRODUCT → USER_FLOW → IA_SCREEN → FEATURE_SPEC → DB_API → ARCHITECTURE → DESIGN_SYSTEM → DEV_RULE 순이다. USER_FLOW와 IA_SCREEN을 FEATURE보다 먼저 생성해야 기능 정의 품질이 높아진다.

## 9단계: 문서 누락 및 충돌 검사기
필수 보강 요소다. 문서 간 충돌과 누락을 자동 감지한다. FEATURE에 로그인 기능이 있는데 USER_FLOW에 로그인 화면이 없거나, 결제 기능이 있는데 DB/API 설계에 반영되지 않은 경우 경고를 띄운다. 이 구조가 명세 완성도를 크게 높인다.

## 10단계: 프로젝트 폴더 자동 생성
모든 문서 생성이 끝나면 개발 가능한 폴더 구조를 자동 생성한다. 루트에는 CORE.md와 env.local을 두고 docs 내부에 나머지 MD 파일을 배치한다. API Vault 연결 정보도 포함한다. 결과물은 개발 도구에 즉시 넣을 수 있는 상태여야 한다.

## 11단계: 개발 툴 투입 모드
개발 툴별 bootstrap prompt 생성 기능을 추가한다. 각 툴의 특성에 맞춰 읽는 순서와 우선순위를 자동 정의한다. 예를 들어 빠른 MVP 중심 툴과 설계 중심 툴은 참조 방식이 다르므로, AI가 툴별 최적 프롬프트를 자동 생성해야 한다.

## 12단계: 개발 후 역동기화 시스템
강력 추천 기능이다. 코드가 변경되면 명세와 싱크가 깨지는 문제를 해결한다. 변경 감지를 통해 “로그인 기능이 추가됐습니다. FEATURE_SPEC을 업데이트할까요?”처럼 문서 갱신을 제안한다. 결과적으로 명세가 죽은 문서가 아니라 살아있는 문서가 된다.

## 13단계: MVP 자동 축소 및 우선순위 엔진
비개발자는 기능 욕심 때문에 처음부터 과도하게 큰 서비스를 설계하는 경우가 많다. 이를 방지하기 위해 AI가 자동으로 “지금 꼭 필요한 기능”과 “나중에 붙여도 되는 기능”을 분리한다. 예를 들어 핵심 가치와 직접 연결되지 않는 기능은 후순위로 밀어낸다. 동시에 MVP(최소 실행 제품), 표준 버전, 확장 버전의 3단계 로드맵을 자동 생성한다. 결과적으로 “일단 돌아가는 제품”을 빠르게 만들 수 있게 한다.

## 14단계: 비용·운영 시뮬레이션 시스템
명세가 어느 정도 완성되면 실제 운영 비용을 예측한다. AI API 사용량, 이미지 생성 비용, DB 저장량, 서버 비용, 사용자 증가 시 확장 비용 등을 시뮬레이션한다. 또한 “혼자 운영 가능”, “월 비용 10만원 이하”, “트래픽 증가 시 위험” 같은 쉬운 언어로 설명한다. 비개발자가 개발 전에 현실성을 판단할 수 있게 만드는 단계다.

## 15단계: 리스크 및 누락 사전 감지 시스템
서비스가 현실에서 부딪힐 문제를 미리 탐지한다. 로그인·결제·개인정보·저작권·외부 API 제한·법적 이슈 등을 자동 점검한다. 예를 들어 회원가입이 있는데 개인정보 처리 흐름이 없거나, 유튜브 데이터를 쓰는데 정책 문제가 있을 경우 경고한다. 동시에 “왜 위험한지”와 “쉬운 해결 방법”을 설명한다.

## 16단계: 개발 태스크 자동 분해 시스템
명세가 끝나면 실제 개발 작업 단위로 자동 분해한다. 단순히 문서를 만드는 것이 아니라 “무엇부터 개발해야 하는지”까지 연결한다. 예를 들어 로그인 만들기, DB 연결하기, 첫 화면 만들기, API 연결하기 같은 작업 리스트를 우선순위와 의존 관계 기반으로 자동 생성한다. 개발 툴이나 AI 에이전트가 그대로 작업 가능한 수준까지 구조화한다.

## 17단계: 배포·운영 전 최종 검수 시스템
개발 완료 직전 마지막 품질 검사 단계다. 로그인 정상 동작 여부, API 키 누락 여부, 모바일 반응형 상태, DB 연결 상태, 환경변수 설정, 오류 처리, SEO 기본값, 결제 흐름, 개인정보 안내 등 실제 서비스 오픈 전에 필요한 체크리스트를 자동 검사한다. “바로 공개 가능”, “테스트 필요”, “치명적 문제 있음”처럼 쉬운 결과 리포트를 제공해 비개발자도 안심하고 배포할 수 있게 만든다.
